Engine Project Summary
Project Overview

This project began as a terminal pseudo-isometric tile game. The original architecture was built around a grid because the terminal interface naturally encouraged that representation.

After moving toward real graphics, the engine has evolved away from grid-based thinking into a true 3D engine.

The engine is written in C using SDL2 for window creation and basic rendering. The goal is to understand and implement core engine systems from first principles rather than immediately relying on existing graphics engines.

The long-term goal is a game combining elements of:

Skyrim-style natural environments
Minecraft-style world freedom
Valheim-style survival and simulation systems

The focus is not only rendering, but coherent world systems:

physics
terrain representation
object systems
world streaming
simulation
Engine Philosophy

The project follows these principles:

Understand the math before abstracting it.
Build systems incrementally.
Avoid complexity until it is justified.
Prefer physically motivated systems over arbitrary rules.
Keep representations meaningful rather than optimizing prematurely.

Physics System

The engine uses continuous 3D physics.

The player currently has:

position
velocity
acceleration
applied forces
mass
maximum force

Physics update:

net force
=
player force
+
gravity
+
drag


Using:

F = ma

velocity += acceleration * dt

position += velocity * dt


Drag:

Fdrag = -k * velocity


The coefficient is tuned for gameplay rather than strict physical accuracy.

Important realization:

Removing movement input does not naturally feel like stopping because humans apply braking forces. A future character controller may generate counter-force when the desired velocity reaches zero.

Coordinate System

The engine uses:

X = left/right
Y = up/down
Z = forward/back


This follows common 3D graphics conventions.

Movement is currently world-relative.

Future improvement:

Convert movement forces into camera-relative movement.

Terrain System
Design Philosophy

Terrain is not intended to be purely procedural.

The intended workflow is:

hand-authored terrain
generated terrain where useful
human refinement afterward

The goal is not to create infinite random worlds. The goal is to create large, detailed worlds with efficient representation.

Chunk System

The term "chunk" is retained from the original voxel/grid-based design, but chunks are no longer voxel containers.

A chunk is a world-management and streaming unit.

A chunk may contain:

terrain representation
objects
entities
simulation state
loading state

Chunks exist to control:

memory usage
streaming
simulation updates
rendering decisions
Terrain Representation

The terrain system is based on layered surface data rather than a voxel volume.

A traditional heightmap stores:

(x,z) → y


The engine instead allows:

(x,z) → multiple surface points


Each surface point contains information such as:

position
normal
material data

Multiple surfaces at the same horizontal location allow:

caves
cliffs
overhangs
tunnels
multiple vertical layers
Surface Detection

Surface normals determine whether geometry represents exposed terrain.

Conceptually:

A surface facing empty space is a visible boundary.

A surface facing inward indicates enclosed terrain.

This allows the terrain system to infer inside/outside relationships without storing every voxel.

The terrain representation is therefore closer to a layered surface field than a voxel grid.

Terrain Complexity

Chunks do not require the same amount of detail everywhere.

Simple areas can use a lightweight representation:

single surface layer
simple height information


Complex areas can store additional surface layers:

multiple surfaces
caves
cliffs
overhangs


Example:

A mountain surface far above an underground cave does not need to actively process the cave representation when the player is nowhere near it.

Terrain detail is loaded based on relevance.

Perspective Projection

Relative position:

object position - camera position


Projection:

screen_x = camera_x / camera_z
screen_y = camera_y / camera_z


Pixel conversion:

pixel_x = center_x + screen_x * focal
pixel_y = center_y - screen_y * focal


Objects behind the camera are clipped or discarded.

Camera Rotation

The camera supports yaw and pitch.

Yaw:

x' = x cos(theta) - z sin(theta)

z' = x sin(theta) + z cos(theta)


Pitch:

y' = y cos(phi) - z sin(phi)

z' = y sin(phi) + z cos(phi)


The camera now operates in true 3D space.

Mesh System

The renderer uses indexed meshes.

A mesh contains:

Switch Formatting

Switches use this style:

  case(TEST):{

    code;

  $}case(OTHER):{
    code;
  $}default:{$}
}


The $ macro is intentional and should not be replaced.

Nested switches follow the same indentation.
