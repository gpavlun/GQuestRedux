Engine Project Summary (Updated)
Project Overview

The project started as a terminal pseudo-isometric tile game, but the architecture was rebuilt into a true 3D engine rather than extending the old grid-based system.

The engine is written in C with SDL2 handling window creation and basic drawing. The goal is to build the rendering and simulation systems from first principles rather than immediately relying on graphics APIs.

The long-term goal is a game somewhere between:

Skyrim-style natural environments
Minecraft-style procedural worlds
Valheim-style survival/simulation systems

The focus is not only visual rendering, but also coherent systems: physics, procedural terrain, world streaming, and simulation.

Core Engine Philosophy

The project follows a few consistent principles:

Understand the math before abstracting it.
Build systems incrementally.
Avoid introducing complexity before it is needed.
Prefer physically motivated systems over arbitrary game logic.

Examples:

Forces instead of directly setting velocity.
Real coordinate systems instead of tile movement.
Manual projection instead of immediately using OpenGL.
Simple meshes before terrain generation.

Drag is currently:

The coefficient is tuned for gameplay rather than strict physical accuracy.

Important realization:
Removing movement input does not naturally feel like stopping because humans apply braking forces. A future character controller may automatically generate counter-force when the desired velocity is zero.

World Coordinates

The engine uses:

X = left/right
Y = up/down
Z = forward/back


This was chosen because it aligns better with common 3D graphics conventions.

Movement input is currently still world-relative. The next major improvement is converting input forces into camera-relative movement.

Terrain Direction

Traditional heightmaps were rejected because they are fundamentally 2.5D.

Problems:

no caves
no overhangs
no arbitrary topology

The long-term terrain idea is a surface-based representation:

Instead of storing every voxel:

terrain volume


store:

surface graph


Nodes contain:

position
surface normal
neighbor connections

The renderer reconstructs geometry from the surface representation.

Benefits:

supports caves
supports overhangs
memory efficient
streamable by chunks

Generation remains an open problem.

Rendering Progress

The renderer is currently built manually with SDL2.

The first milestone was achieved:

A 3D point can now become a projected screen position.

The basic projection:

relative position:

object - camera


screen:

x = camera_x / camera_z
y = camera_y / camera_z


Then:

pixel_x = center_x + x*focal
pixel_y = center_y - y*focal


Objects behind the camera are discarded.

This produced working perspective:

objects move correctly
objects scale with distance
camera-relative positioning works
Camera Rotation

The camera now supports yaw and pitch.

The pipeline became:

world coordinates

      |
      v

camera-relative coordinates

      |
      v

yaw rotation

      |
      v

pitch rotation

      |
      v

perspective projection

      |
      v

screen coordinates


Yaw:

x' = x cos(theta) - z sin(theta)

z' = x sin(theta) + z cos(theta)


Pitch:

y' = y cos(phi) - z sin(phi)

z' = y sin(phi) + z cos(phi)


The camera now moves through a true 3D space.

Mesh System

The renderer moved from drawing rectangles to indexed meshes.

A mesh contains:

vertices
edges
triangles

Vertices are stored once.

Triangles reference vertex indices.

Example:

vertices:
0
1
2
3

triangle:
0,1,2


This is the basis for real 3D models.

Wireframe Rendering

The engine can now:

transform mesh vertices
project them
draw edges between vertices

This was the first moment the engine became a real 3D renderer.

A major debugging issue was discovered:

Screen-space points were stored as unsigned integers.

This broke negative coordinates because:

The fix:

Screen coordinates must use signed types.

Overall State

The project has crossed an important boundary. It started as a tile-based experiment and now has the foundations of a real 3D engine:

continuous physics
real camera space
perspective projection
mesh geometry
triangle rasterization

The next challenges are no longer "how do I make 3D math work?" but the actual problems of building a renderer: visibility, object transforms, depth, terrain, and scale.
